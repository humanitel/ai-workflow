Installation & Setup

Everything you need to go from zero to a fully configured Claude Code environment with plugins, skills, agents, and MCP servers.

1. Install Claude Code

# Install the CLI globally
npm install -g @anthropic-ai/claude-code

# Verify installation
claude --version

# Start Claude Code in any project directory
cd your-project/
claude

2. Configure Settings

Claude Code settings live at ~/.claude/settings.json. Key things to configure:

{
  "permissions": {
    "allow": [
      "Bash(ls:*)",
      "Bash(git:*)",
      "Bash(npm run:*)",
      "Bash(npx:*)"
    ]
  },
  "plugins": {
    "enabled": [
      "superpowers@claude-plugins-official",
      "frontend-design@claude-plugins-official"
    ]
  }
}
Tip: Symlink your settings from a dotfiles repo to keep them consistent across machines: ln -s ~/dotfiles/claude/settings.json ~/.claude/settings.json

3. Install Plugins

Plugins bundle multiple skills together. The two essential plugins:

  • Superpowers — 14 skills for planning, development, debugging, code review, and shipping workflows
  • Frontend Design — production-grade UI generation with distinctive aesthetics

Plugins are configured in settings.json under the plugins.enabled array. Claude Code installs and caches them automatically at ~/.claude/plugins/cache/.

4. Install Custom Skills

Skills are individual workflow definitions stored at ~/.claude/skills/. You can create them directly or symlink from a shared repo:

# Symlink skills from a shared dotfiles repo
ln -s ~/dotfiles/claude/skills/* ~/.claude/skills/

# Or clone a skills repo
git clone git@github.com:your-org/claude-skills.git ~/.claude/skills/shared

Each skill is a directory containing a skill.md (or similar) file that defines when and how the skill activates.

5. Configure MCP Servers

MCP (Model Context Protocol) servers extend Claude with external tool integrations. Configure them in settings.json:

  • Pencil — Design editor for visual design files
  • Chrome (claude-in-chrome) — Browser automation, QA testing, form filling
  • Linear — Issue tracking, project management
  • Excalidraw — Diagrams and visual collaboration

6. Set Up CLAUDE.md Files

CLAUDE.md files provide project-scoped instructions that Claude follows automatically. Place them at:

  • ~/.claude/CLAUDE.md — Global instructions (all projects)
  • your-project/CLAUDE.md — Project root instructions
  • your-project/apps/web/CLAUDE.md — Sub-directory specific instructions
CLAUDE.md is how you teach Claude about your codebase — architecture, conventions, commands, patterns. Think of it as onboarding docs for your AI pair programmer.

7. Configure Memory

Claude has a persistent memory system at ~/.claude/projects/{project}/memory/. Memory carries across conversations. Types:

  • user — Your role, preferences, expertise level
  • feedback — Corrections and guidance you've given Claude
  • project — Ongoing work context, decisions, deadlines
  • reference — Pointers to external systems (Linear projects, Grafana dashboards)

Memory builds up automatically as you work. You can also explicitly say "remember that..." to save something.

How It All Fits Together

The mental model for understanding Claude Code's extensibility layers.

CLAUDE.md

Project-scoped instructions Claude follows automatically. Defines architecture, conventions, and commands. Loaded at conversation start.

Skills

Reusable workflow definitions. Auto-trigger based on context or invoked via /skill-name. Define HOW to approach tasks.

Plugins

Bundles of skills distributed as packages. Installed once, available everywhere. Superpowers plugin includes 14 development workflow skills.

Agents

Subprocesses Claude spawns to EXECUTE work. Run in parallel or background. Each agent type has specialized tools and capabilities.

MCP Servers

External tool integrations via Model Context Protocol. Browser automation, design tools, project management. Extend what Claude can interact with.

Memory

Persistent file-based knowledge across conversations. Stores your preferences, project context, and corrections so Claude improves over time.

Hooks

Shell commands that execute in response to Claude Code events. Automate behaviors like "run lint before every commit" or "notify on task completion."

Typical Task Flow

You describe a task → Skills auto-trigger → Claude plans the approach → Agents execute in parallel → MCP tools interact externally → Memory stores context
Key insight: Skills define the workflow (brainstorm → plan → execute → review). Agents handle the execution (parallel research, isolated code changes). MCP servers provide the tools (browser, design editor, issue tracker). CLAUDE.md provides the context (project architecture, conventions).

Workflows

Scenario-based guides for common tasks. Each is a self-contained mini-runbook.

Getting Started

1 Understanding the Basics

▸
essentials

When to use

You're new to Claude Code and want to understand how skills, slash commands, and auto-triggering work.

What happens

Claude Code reads your CLAUDE.md files at conversation start, loads available skills, and watches for context clues to auto-trigger relevant workflows.

Key concepts

  • Auto-triggering: Skills activate automatically based on what you're doing. Start debugging? The systematic-debugging skill kicks in. Writing a feature? brainstorming activates first.
  • Slash commands: Type /skill-name to explicitly invoke a skill. Example: /ship, /review, /retro
  • CLAUDE.md layering: Global ~/.claude/CLAUDE.md applies everywhere. Project-level adds project-specific context. Sub-directory files add more granular instructions.
  • Memory persistence: Claude remembers your preferences, past corrections, and project context across conversations.

Example

# Just start Claude in your project
claude

# Or invoke a specific skill
> /brainstorm add a notification system
> /ship
> /review

2 Understanding Agents

▸
agents essentials

When to use

You want to understand how Claude spawns and uses subagents for parallel work.

What happens

Claude can launch specialized agents (subprocesses) that work autonomously on tasks. Each agent type has specific tools and capabilities. Agents can run in the foreground (Claude waits for results) or background (Claude continues working).

Agent types

  • general-purpose: Multi-step tasks, code search, complex operations. Has access to all tools.
  • Explore: Fast codebase exploration. Searches for files, keywords, and answers codebase questions. Three levels: quick, medium, very thorough.
  • Plan: Software architecture and implementation planning. Identifies critical files and trade-offs.
  • code-reviewer: Reviews completed work against plans and coding standards.
  • vercel_ai_sdk: Expert in Vercel AI SDK, builds AI-powered React components.
  • prompter: Crafts and compares system prompts (system mode for writing, view mode for visual comparison).
  • linear_pm: Manages Linear projects — tickets, sprints, issues, executive summaries.

Foreground vs Background

  • Foreground (default): Claude waits for the agent to finish. Use when the result informs the next step.
  • Background: Claude continues working while the agent runs. Use for independent tasks.

Worktree isolation

Agents can run in an isolated git worktree — a temporary copy of the repo. Changes stay isolated until explicitly merged. Great for parallel feature work.

3 Starting a New Feature

▸
brainstorming writing-plans executing-plans using-git-worktrees

When to use

You have a feature to build and want Claude to help from ideation through implementation.

What happens

The Superpowers plugin enforces a disciplined workflow:

Brainstorm → Write Plan → Execute Plan → Review
  1. Brainstorming: Claude asks clarifying questions one at a time, proposes 2-3 approaches with trade-offs, and presents a design for your approval.
  2. Writing Plans: The approved design becomes a detailed implementation plan with ordered steps, file paths, and test coverage.
  3. Executing Plans: Claude works through the plan step by step, with review checkpoints. Can use parallel agents for independent tasks.
  4. Git Worktrees: Optionally isolate the work in a git worktree so your main branch stays clean.

How to invoke

# Just describe your feature — brainstorming auto-triggers
> I need to add a notification system for overdue invoices

# Or start in a worktree for isolation
> /using-git-worktrees
> I need to add a notification system for overdue invoices

5 Designing with Pencil

▸
Pencil MCP

When to use

You want to create, edit, or review visual designs using .pen files — screens, components, dashboards, landing pages, mobile apps, or slide decks.

What happens

Pencil is an MCP-powered design editor that Claude interacts with directly. You describe what you want, and Claude builds it visually in a .pen file — inserting nodes, styling them, generating images, and validating with screenshots.

Typical workflow

  1. Open or create a file: Claude opens an existing .pen file or creates a new one
  2. Get guidelines: Claude fetches design guidelines for the task type (web-app, mobile-app, landing-page, slides, etc.)
  3. Get a style guide: Claude pulls a style guide for visual consistency and inspiration
  4. Design iteratively: Claude uses batch_design to insert, update, copy, replace, and arrange nodes — up to 25 operations per call
  5. Validate visually: Claude takes screenshots (get_screenshot) to check layout and appearance
  6. Refine: You review and request changes; Claude adjusts using snapshot_layout to understand positioning

Key operations

  • batch_get — Search and read nodes by patterns or IDs
  • batch_design — Insert (I), Copy (C), Replace (R), Update (U), Delete (D), Move (M), Generate image (G)
  • get_screenshot — Visual validation of any node
  • snapshot_layout — Inspect computed layout rectangles for precise positioning
  • get_variables / set_variables — Manage design tokens and themes
  • export_nodes — Export to PNG, JPEG, WEBP, or PDF
  • find_empty_space_on_canvas — Find room for new elements

How to invoke

# Claude uses Pencil tools when working with .pen files
> Open the dashboard design and add a revenue chart
> Create a new mobile app design for the onboarding flow
> Update the color scheme to use our brand colors
> Export the login screen as a PNG

# Design system work
> Set up variables for our color palette and typography
> Apply the dark theme across all screens
Important: .pen file contents are encrypted and can only be accessed via Pencil MCP tools. Never use Read or Grep to read .pen files directly.

Related

  • Writing Frontend Code — implement designs as code after visual approval
  • MCP Servers: Pencil — full tool reference

7 Running Tests (TDD)

▸
test-driven-development testing-guidelines

When to use

Implementing any feature or bugfix. The TDD skill activates before you write implementation code.

What happens

Claude follows a strict red-green-refactor cycle:

  1. Red: Write a failing test that describes the desired behavior
  2. Green: Write the minimum code to make the test pass
  3. Refactor: Clean up while keeping tests green

Key behaviors

  • Emphasizes integration tests over unit tests with mocks
  • Uses real-world fixtures and factory patterns
  • Ensures regression coverage for bug fixes

8 Debugging an Issue

▸
systematic-debugging grafana-logs

When to use

Any bug, test failure, or unexpected behavior — before proposing fixes.

What happens

The systematic-debugging skill forces a disciplined approach:

  1. Observe: Reproduce the issue, gather logs, read error messages carefully
  2. Hypothesize: Form specific, testable theories about the root cause
  3. Test: Validate each hypothesis with evidence
  4. Fix: Address the root cause, not symptoms
  5. Verify: Confirm the fix resolves the original issue

For production issues

The grafana-logs skill connects to your Grafana/Loki instance to query production logs directly from Claude Code.

# Auto-triggers on bugs
> The payment webhook is returning 500 errors intermittently

# For production log investigation
> /grafana-logs check for payment webhook errors in the last hour

When to use

You have 2+ independent tasks that can be worked on without shared state or sequential dependencies.

What happens

Claude launches multiple agents simultaneously, each working on an isolated task. Agents can run in git worktrees for full code isolation.

Examples

  • Research 3 different API approaches in parallel
  • Implement independent components simultaneously
  • Run tests and lint in parallel while reviewing code
  • Explore multiple parts of a codebase at once

How it works

# Claude decides to parallelize automatically
# Or you can hint at it:
> Implement the notification service and the email template
> system in parallel — they're independent
Tip: Parallel agents work best when tasks are truly independent. If task B depends on task A's output, they must run sequentially.

22 Managing Memory

▸
core system

When to use

You want Claude to remember something for future conversations, or you want to review/prune what it knows.

How memory works

  • Memory files live at ~/.claude/projects/{project}/memory/
  • An index file (MEMORY.md) tracks all memories
  • Four types: user (role/preferences), feedback (corrections), project (work context), reference (external pointers)
  • Claude loads relevant memories at conversation start

How to use

# Save something to memory
> Remember that I prefer integration tests over unit tests with mocks

# Claude saves feedback from corrections automatically
> No, don't use styled-components — we've migrated to Tailwind
# (Claude remembers this for next time)

# Ask Claude to recall
> What do you remember about my testing preferences?

# Ask Claude to forget
> Forget the memory about the deadline — it's been extended
Tip: Memory is most powerful for feedback. When you correct Claude, it saves that correction and avoids the same mistake in future conversations.

23 Troubleshooting

▸
essentials

When to use

Claude is stuck, using the wrong skill, giving bad answers, or not behaving as expected.

Common issues & fixes

  • Wrong skill: Be more specific. Say "don't use brainstorming, just write the code" or "skip the planning phase."
  • Stuck in a loop: Say "stop" or "try a different approach."
  • Forgot context: Long conversations get compressed. Re-state important context, or use memory: "Remember that we're using the v2 API."
  • Skills not loading: Check ~/.claude/settings.json for plugin configuration.
  • MCP server not responding: Check that the extension/server is running.
  • Permission denied: Add the command to settings.json permissions, or approve when prompted.

Power user tips

  • Use /help to see available commands
  • Give feedback to correct behavior: "Don't do X, instead do Y." Claude saves this to memory.
  • For complex tasks, let the brainstorming skill run — it prevents wasted implementation work
  • If Claude's response is too verbose, say "be more concise" — it'll remember
